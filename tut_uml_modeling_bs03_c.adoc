== モデルとコードの対応づけ

[.lead]
プログラムがどのように動作するのか検討するときには、振る舞いのモデルを使います。
振る舞いのモデルを作成する前に、構造のモデルと振る舞いのモデルがどのようなコードとして実現されるのかについて、このチュートリアルで用いる方式を決めておきましょう。
このことを「モデルとコードの対応づけ」または「実装方式の設計」と呼ぶことにします。

TIP: このような対応づけを「モデル変換ルール」と呼ぶこともあります。また、開発プロセスのなかでこの対応づけを検討する工程を「方式設計」と呼ぶ人もいます。

モデルとコードの対応づけを決めてそれを前提にして設計する（モデルを作る）ことは、実装に依存する情報と依存しない情報の分離を促します。
また、このような考え方は、実装に依存しないモデルを作る上でとても重要で、実務的なモデルを作成する際にも大いに役立ちます。


=== 対応づけ検討用モデルの作成


このチュートリアルでは実装にC言語を使うことにしました。
サンプルモデルとサンプルモデルに対応するC言語プログラムを作ってみることで、対応づけの方式の基本的な考え方を決めておこうというわけです。

.この章の役割についてもう少し
[sidebar]
--
この章で検討したいことは、ボウリングのスコアに関するモデルの問題領域とは直接の関係はありません。

しかし、モデルとコードの対応づけを考えるための問題領域をボウリングのスコアに関する問題領域と分けて考えておけば、使用するプログラミング言語が異なる場合にも、対応づけのための問題領域を差し替えることで、ボウリングスコアのためのモデルを転用しやすくなります。
そんなわけで、別の問題として分けて検討するために、この章を用意してあります。

もし、この章の役割がピンとこない、実装についての議論にはあまり関心がないのであれば、最初はこの章を読み飛ばしてみてもかまいません。

しかし、後の章へ進むと、実装する人がモデルを使う際には、モデルとコードを対応づける必要があり、ここで実装方式を決めておかなければ、実装者がそのことについて考える必要がある（特定の実装者に依存しないよう設計しておくべきことを実装する各人に委ねてしまっている）ことに気づくでしょう。
そのときは、この章に戻って、どうして実装方式の設計が必要なのか考えてもらえたらと思います。
--

===== Sampleクラスの最初のサンプルコード

対応づけルールの1つ目として、 <<sample_class01>> を表したC言語のヘッダファイルとCファイルのセットで書きたいと決めることにします。

まず、ヘッダファイル `sample.h` については、<<c_sample01_h>> のように対応づけます。

[[c_sample01_h]]
.【C言語】「Sample」クラスを表すC言語のコード（ヘッダファイル）
[source,c]
----
/* sample.h - Sampleクラスの定義 */
#ifndef SAMPLE_H
#define SAMPLE_H

#include <stdbool.h>

typedef struct {  // <1>
    bool attr_a;  // <2>
    bool attr_b;  // <2>
    /* 状態変数は後で追加 */
} Sample;

/* メソッドにあたる関数のプロトタイプ宣言 */
void sample_init(Sample *sample);  // <3>
void sample_play(Sample *sample);  // <4>

#endif /* SAMPLE_H */
----
<1> 構造体「Sample」の定義（UMLのクラスに対応）
<2> このクラスのメンバー変数（UMLの属性に対応）
<3> 初期化関数（コンストラクタに相当する関数）
<4> メソッド「play」に対応する関数


.C言語でのクラスの表現
[sidebar]
--
C言語はオブジェクト指向言語ではありませんが、構造体(`struct`)を使ってクラスの概念を表現できます。

* *クラス* → 構造体(`struct`)
* *属性* → 構造体のメンバー変数
* *メソッド* → 構造体へのポインタを第1引数に取る関数
* *コンストラクタ* → `xxx_init()` 関数
* *カプセル化* → ヘッダーファイルとソースファイルの分離、static関数
--

そして、実装ファイル`sample.c`は <<c_sample01_c>> のようになります。

[[c_sample01_c]]
.【C言語】「Sample」クラスの実装
[source,c]
----
/* sample.c - Sampleクラスの実装 */
#include "sample.h"
#include <stdio.h>

/* Sample構造体の初期化 */
void sample_init(Sample *sample) {  // <1>
    sample->attr_a = true;  // <2>
    sample->attr_b = true;
    /* 状態の初期化は後で追加 */
}

/* playメソッドの実装 */
void sample_play(Sample *sample) {  // <3>
    /* 振る舞いは後で実装 */
    printf("play called\n");
}
----
<1> 初期化関数の実装
<2> メンバー変数の初期化
<3> playメソッドの実装

.実装方式の策定も設計の一部
[sidebar]
--
C言語に限ったことではありませんが、モデルとコードの対応づけ（実装方式）は、決まったものがあるわけではありません。重要なのは、自分たちがチームで決まった方式を決めておき、モデルの作成（設計時）にもその実装方式を前提に設計し、実装時にもその実装方式に従って実装することです。

たとえば、「テスト駆動開発による組み込みプログラミング」 <<embtdd>> では、「SOLIDの原則をCモジュールに提要した設計モデル」として次のようなモデルを紹介しています。

シングルインスタンスモジュール（Single-instance module）:: モジュールの内部構造をカプセル化するが、モジュールのインスタンスがひとつであることを前提とする。
マルチインスタンスモジュール（Multi-instance module）:: モジュールの内部構造をカプセル化して、モジュールのインスタンスを複数作れるようにする。
動的インターフェイス（Dynamic interface）:: モジュールのインターフェイス関数を実行時に割り当てられるようにする。
型による動的インターフェイス（Pre-type dynamic module）:: 同じインターフェイスを持つモジュールの型が複数あり、それぞれが独自のインターフェイス関数を持てるようにする。

この分類からみると、このチュートリアルが採用した実装方式は、「マルチインスタンスモジュール」に相当することがわかるでしょう。

さらに、クラス、シーケンス、状態遷移といった設計上の概念を、実装時にどのように実現するのかを決めておくこともまた重要な設計上の決定になります。設計の方針ともなるので、「実装方式の決定はアーキテクチャ設計である」と捉える場合もあります。
--

=== 変換ルールを使ったコードを確認する

ステートマシン図をC言語で実装する方法を確認しましょう。

==== 状態の定義

ステートマシン図の「状態」をC言語で表現するには、`enum`（列挙型）を使います。

[[c_state_enum]]
.状態の定義
[source,c]
----
typedef enum {
    ST0,  // <1>
    ST1,
    ST2
} SampleState;  // <2>
----
<1> 各状態を列挙型の値として定義
<2> 状態を表す型名

==== イベントの定義

ステートマシン図の「イベント」も`enum`で定義します。

[[c_event_enum]]
.イベントの定義
[source,c]
----
typedef enum {
    EV1,  // <1>
    EV2,
    EV3
} SampleEvent;  // <2>
----
<1> 各イベントを列挙型の値として定義
<2> イベントを表す型名

==== Sample構造体に状態変数を追加

構造体に現在の状態を保持する変数を追加します。

[[c_sample_with_state]]
.状態変数を持つSample構造体
[source,c]
----
typedef struct {
    bool attr_a;
    bool attr_b;
    SampleState state;  // <1>
} Sample;
----
<1> 現在の状態を保持する変数

==== 状態遷移処理の実装

状態遷移の処理は、現在の状態とイベントに応じて処理を振り分ける関数として実装します。

[[c_state_transition]]
.状態遷移処理の基本構造
[source,c]
----
void sample_play(Sample *sample, SampleEvent evt, int param) {
    switch (sample->state) {  // <1>
        case ST0:  // <2>
            st0_proc(sample, evt, param);  // <3>
            break;
        case ST1:
            st1_proc(sample, evt, param);
            break;
        case ST2:
            st2_proc(sample, evt, param);
            break;
    }
}
----
<1> 現在の状態で場合分けする
<2> 状態「ST0」の場合
<3> その状態での処理を行う関数を呼び出す

==== 状態ごとの処理関数

各状態での処理は、イベントに応じて適切な処理を行います。

[[c_st0_proc]]
.ST0状態での処理
[source,c]
----
static void st0_proc(Sample *sample, SampleEvent evt, int param) {
    switch (evt) {  // <1>
        case EV1:  // <2>
            if (gd1(sample)) {  // <3>
                act1(sample, evt, param);  // <4>
                sample->state = ST1;  // <5>
            } else {
                printf("Event ignored due to guard condition\n");  // <6>
            }
            break;
        case EV3:  // <7>
            act3(sample, evt, param);
            sample->state = ST0;
            break;
        default:
            /* その他のイベントは無視 */
            break;
    }
}
----
<1> イベントで場合分けする
<2> イベントが「EV1」の場合
<3> ガード条件による判定
<4> アクション「act1」の呼び出し
<5> 次の状態「ST1」への遷移
<6> ガード条件が偽だったときの処理
<7> イベントが「EV3」の場合

[TIP]
.C言語での関数のスコープ
--
C言語では、`static`キーワードを使って関数のスコープを制限できます：

* `static`のない関数: 他のファイルから呼び出し可能（public相当）
* `static`付き関数: 同じファイル内からのみ呼び出し可能（private相当）

`st0_proc`などの内部処理関数は`static`にして、外部から直接呼び出せないようにします。
--

==== 選択疑似状態の実装

選択疑似状態は、C言語では`if-else`文で実装します。

[[c_st1_proc]]
.ST1状態での処理（選択疑似状態を含む）
[source,c]
----
static void st1_proc(Sample *sample, SampleEvent evt, int param) {
    switch (evt) {
        case EV2:  // <1>
            act2(sample, evt, param);  // <2>
            if (gd2(sample)) {  // <3>
                sample->state = ST2;  // <4>
            } else {
                act3(sample, evt, param);  // <5>
                sample->state = ST0;  // <6>
            }
            break;
        case EV3:  // <7>
            act3(sample, evt, param);
            sample->state = ST0;
            break;
        default:
            break;
    }
}
----
<1> イベントが「EV2」の場合
<2> アクション「act2」の呼び出し
<3> 選択疑似状態におけるガード条件による判定
<4> 次の状態「ST2」への遷移
<5> 選択疑似状態からの遷移におけるアクション「act3」の呼び出し
<6> 次の状態「ST0」への遷移
<7> イベントが「EV3」の場合

==== アクションとガード条件の実装

アクションとガード条件は、それぞれ関数として実装します。

[[c_actions_guards]]
.アクションとガード条件の実装例
[source,c]
----
/* ガード条件の実装 */
static bool gd1(const Sample *sample) {
    return sample->attr_a;  // <1>
}

static bool gd2(const Sample *sample) {
    return sample->attr_b;
}

/* アクションの実装 */
static void act1(Sample *sample, SampleEvent evt, int param) {
    printf("Action act1 executed\n");  // <2>
    /* 実際の処理 */
}

static void act2(Sample *sample, SampleEvent evt, int param) {
    printf("Action act2 executed\n");
    /* 実際の処理 */
}

static void act3(Sample *sample, SampleEvent evt, int param) {
    printf("Action act3 executed\n");
    /* 実際の処理 */
}
----
<1> ガード条件は真偽値を返す関数として実装
<2> アクションは必要な処理を行う関数として実装

=== まとめ

この章では、UMLのクラス図とステートマシン図をC言語のコードにどのように対応づけるかを学びました。

.UMLのモデルとC言語のコードの対応関係
* *クラス* → `struct`（構造体）
* *属性* → 構造体のメンバー変数
* *操作* → 構造体へのポインタを引数に取る関数
* *状態* → `enum`型の値
* *イベント* → `enum`型の値
* *状態遷移* → `switch`文による処理の振り分け
* *ガード条件* → 真偽値を返す関数
* *アクション* → 処理を行う関数
* *選択疑似状態* → `if-else`文

次の章からは、この対応づけルールを使って、実際のボウリングスコアシステムのモデルを作成し、C言語で実装します。
