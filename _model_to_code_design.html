第3章 モデルとコードの対応づけ

プログラムがどのように動作するのか検討するときには、振る舞いのモデルを使います。
振る舞いのモデルを作成する前に、構造のモデルと振る舞いのモデルがどのようなコードとして実現されるのかについて、このチュートリアルの開発で用いる方式を決めておきましょう。
このことを「モデルとコードの対応づけ」または「実装方式の設計」と呼ぶことにします。

ヒント
 このような対応づけを「モデル変換ルール」と呼ぶこともあります。また、開発プロセスのなかでこの対応づけを検討する工程を「方式設計」と呼ぶ人もいます。
この章の役割についてもう少し

この章で検討したいことは、ボウリングのスコアに関してどのようなモデルを作るのか、という話題から少し離れています。 モデルをどのように実装するのかということは、モデルの表し方とコードの表し方の間の関係を定めるものであって、ボウリングのスコアをどのようなモデルとして表すかということとは独立している話題だからです。

もし、この章の役割がピンとこないとか、実装についての議論にはあまり関心がないのであれば、最初はこの章を読み飛ばしてみてもかまいません。 後の章へ進めば、実装する人がモデルを使う際には、モデルとコードを対応づける必要があり、事前に実装方式を決めておかなければ、実装者が実装方法について考える（実装時に「どのように作るのか」を設計する）必要があることに気づくでしょう。 そのときは、この章に戻って、どうして実装方式の設計が必要なのか考えてもらえたらと思います。

3.1. なぜ対応づけの検討が必要なのか

ソフトウェア開発でよく目にする事態として、設計と実装が乖離してしまい、頼れるのは動いているコードだけというものがあります。 このとき、多く人が「設計書がないから」「設計書が更新できていないから」と説明します。 このことについて、もう少し詳しく考えてみましょう。

ソフトウェアの開発に使う設計書には次のような2つの役割があります。

設計書の役割

  • 「何を作るか」 について、構造と振る舞いをもって示す。

  • 「どのようにして作るか」 について、設計書と実装の対応づけをもって示す。

「何を作るか」の設計書は、どんな機能を提供し、それがどのような動作・振る舞いとなっているかを示す設計書です。 例えば、「機能的な要求の発見・獲得→要求の分析→外部設計→内部設計」という開発プロセスがあるとき、「要求分析のユースケース図」、「外部設計のクラス図・シーケンス図」、「内部設計のクラス図・ステートマシン図」などを作成します。 これらが「何を作るか」を示すための設計書です。

では、「どのようにして作るか」の設計書とはどんなものでしょうか。 これは、例えば、設計書にクラスがあったらC言語の構造体やRubyのクラスに対応づけるとか、状態遷移は2次元配列を使って遷移表として実装しようとか…そういった、設計書に記載した要素に対して、実装するときにコードのどの部分にどのように反映するかを示した設計書です。

先に挙げた「何を作るか」の設計書には、「どのような実装方法で実装するのか」は書いてありません。 しかし、「どのようにして作るか」の設計書がないと、その側面は実装者がコードを作成しながら考えることになります。 そうなると、「どのようにして作るか」は、実装者ごとに考えることになり、また、同じ実装社でも実装する毎に異なってきます。しかも、その考えはコードに埋め込まれ、明文化されません。 設計書として明文化されていないならば、実装に不具合があって設計書に戻ったとしても、検討すべき情報がそこにはないということになります かくして、設計と実装の乖離ができあがるというわけです。

注記
 「モデル駆動開発（MDD）」という開発手法に、「モデル・タイプマッピング」と呼ばれる、2つのモデルの間の対応づけについての考え方があります。モデルとコードの対応づけは、タイプマッピングの一種とみなすこともできます。

3.3.2. Sampleクラスの最初の対応づけ

対応づけルールの1つ目として、次のように書きたいと決めることにします。 操作や属性は公開範囲（スコープ）を設定できますが、ここでは演習の簡単のために、属性はプライベート、操作はパブリックということにします。

UMLのクラスとC言語のクラスの対応づけ（1）

  • UMLのクラス名を、C言語の typedef で構造体型として定義する。

  • UMLのクラスの属性を、クラスを割り当てた構造体の中のメンバ変数とする。

  • UMLのクラスの操作を、最初の引数がクラスを割り当てた構造体型へのポインタになっている関数とする。

  • クラスのインスタンスを作成するための初期化関数（コンストラクタ）として init の付く関数を用意する。

C言語でのクラスの表現

C言語はオブジェクト指向言語ではありませんが、構造体(struct)を使ってクラスの概念を表現できます。

このチュートリアルが採用した実装方式では、次のような対応づけになっています。

  • クラス → 構造体(struct)

  • 属性 → 構造体のメンバー変数

  • メソッド → 構造体へのポインタを第1引数に取る関数

  • コンストラクタ → xxx_init() 関数

  • カプセル化 → ヘッダーファイルとソースファイルの分離、static関数

たとえば、シングルインスタンス・モジュール（ コラム 2 で紹介）では、次のような対応づけを使うこともできます。

  • クラス → ファイルセット（クラス名のわかる .c と .h ）

  • 属性 → そのファイルの static なグローバル変数

  • メソッド → そのファイルで定義している関数

    • クラス名のわかるプレフィックスをつける

    • パブリックなメソッドは、ヘッダーファイルに関数のプロトタイプ宣言を追加

  • コンストラクタ → クラス名_init() 関数を定義

  • カプセル化 → staticなグローバル変数、パブリックなプロトタイプ宣言の活用

3.3.3. Sampleクラスの最初のサンプルコード

対応づけルールの1つ目として、 図 3.3 を表したCのヘッダーファイル（ .h ファイル）と実装ファイル（ .c ファイル）のセットで書きたいと決めることにします。

まず、ヘッダーファイル sample.h については、リスト 3.1 のように対応づけます。

リスト 3.1 sample.h: 「Sample」クラスを表すC言語のヘッダーファイル
#ifndef SAMPLE_H
#define SAMPLE_H

#include <stdio.h>
#include <stdbool.h>

typedef struct {  // (1)
    bool attr_a;  // (2)
    bool attr_b;  // (2)
    /* 状態変数は後で追加 */
} Sample;

/* メソッドにあたる関数のプロトタイプ宣言 */
void sample_init(Sample *sample);  // (3)
void sample_play(Sample *sample);  // (4)

#endif /* SAMPLE_H */

  1. 構造体「Sample」の定義（UMLのクラスに対応）。

  2. このクラスのメンバー変数（UMLの属性に対応）。

  3. 初期化関数（コンストラクタに相当する関数）。

  4. 操作「play」に対応する関数。

そして、実装ファイル sample.c は リスト 3.2 のようになります。

リスト 3.2 sample.c: 「Sample」クラスの実装（.cファイル）
#include "sample.h" // (1)

/* Sample構造体の初期化（インスタンスの作成） */
void sample_init(Sample *sample) {  // (2)
    sample->attr_a = true;  // (3)
    sample->attr_b = true;  // (3)
    /* 状態の初期化は後で追加 */
}

/* playメソッドの実装 */
void sample_play(Sample *sample) {  // (4)
    /* 振る舞いは後で実装 */
    printf("play called attr_a:%d\n", sample->attr_a);
}

  1. 実装するクラスのヘッダーファイルのインクルード。

  2. 初期化関数の実装。

  3. メンバー変数の初期化。

  4. playメソッドの実装。

Sampleクラスを利用する場合には、Sampleクラスのインスタンスを作り、その操作（メソッド）を呼び出します（ リスト 3.3 ）。

リスト 3.3 Sampleクラスを使うコード例
#include "sample.h" // (1)

int main(void) {
    Sample smpl01;               // (2)
    sample_init(&smpl01);        // (3)
    sample_play(&smpl01);        // (4)
}

  1. 使用するクラスのヘッダーファイルのインクルード。

  2. Sampleクラスのインスタンスに使う変数 smpl01 を定義。

  3. smpl01 の初期化。

  4. smpl01 のメソッド play の呼び出し。

ヒント
 インスタンスに使う変数は、より一般的には malloc などで動的に確保する方法が使われます。この場合、メソッドとなる関数には確保した領域のポインタを渡すことや、使用後に free を使って領域を開放することに注意します。
注記
 この段階のサンプルコードは、 codes/sample01 ディレクトリに用意してあります。
実装方式の策定も設計の一部

C言語に限ったことではありませんが、モデルとコードの対応づけ（実装方式）は、決まったものがあるわけではありません。重要なのは、自分たちがチームで決まった方式を決めておき、モデルの作成（設計時）にもその実装方式を前提に設計し、実装時にもその実装方式に従って実装することです。

たとえば、「テスト駆動開発による組み込みプログラミング」 [embtdd] では、「SOLIDの原則をCモジュールに提要した設計モデル」として次のようなモデルを紹介しています。

シングルインスタンスモジュール（Single-instance module）

モジュールの内部構造をカプセル化するが、モジュールのインスタンスがひとつであることを前提とする。

マルチインスタンスモジュール（Multi-instance module）

モジュールの内部構造をカプセル化して、モジュールのインスタンスを複数作れるようにする。

動的インターフェイス（Dynamic interface）

モジュールのインターフェイス関数を実行時に割り当てられるようにする。

型による動的インターフェイス（Pre-type dynamic module）

同じインターフェイスを持つモジュールの型が複数あり、それぞれが独自のインターフェイス関数を持てるようにする。

この分類からみると、このチュートリアルが採用した実装方式は、「マルチインスタンスモジュール」に相当することがわかるでしょう。

さらに、クラス、シーケンス、状態遷移といった設計上の概念を、実装時にどのように実現するのかを決めておくこともまた重要な設計上の決定になります。設計の方針ともなるので、「実装方式の決定はアーキテクチャ設計である」と捉える場合もあります。

3.4. 振る舞いのモデルを作成する

このサンプルにおいては、いま追加した操作「play」が、「Sample」クラスの振る舞いを提供しているとします。 しかし、クラス図に描いた「Sample」クラスに操作「play」を追加しただけでは、その処理内容はわかりません。 どこかに「play」の処理内容を表した図が必要です。

そこで、「Sample」クラスに対して、操作「play」の振る舞いを表すステートマシン図を追加します。 ここで、図を追加する対象を「Sample」クラスにしているのは、このステートマシン図が「Sample」クラスの振る舞いのモデル図であるとわかるようにするためです。

  • 右クリックしてポップアップメニューを表示する（ 図 3.4 ）。

  • 「図の追加＞ステートマシン図」を選択すると、ステートマシン図が追加される。

図 3.4 「Sample」クラスにステートマシン図を追加する

  • プロパティーから図の名前を編集し、「Sampleのplayのステートマシン図」とする（ 図 3.5 ）。

  • ダイアグラムエディタのタイトルやタブにも図の名前が反映される。

3.4.4. 状態・イベント・状態遷移・アクション

ステートマシン図で処理を表す時に登場する要素について、簡単に整理しておきましょう。

状態とイベント

ステートマシン図において、状態は「イベント」の発生を待っているところです。 多くの場合、イベントは、そのクラス自身の操作では処理を先に進められなくなるようなできごとです。 たとえば、外部からの入力（操作待ちや受信待ちなど）や、一定の時間経過などがイベントの候補になります。

状態遷移

状態遷移は、ある状態において待っていたイベントが発生して、別の状態へ移ることを表します。 遷移先が2つ以上ある状態では、各々の状態遷移で待っているイベントは異なっている必要があります。 それは、同じイベントで別の状態へ遷移する状態遷移が複数あると、どちらへ遷移すべきかをイベントの違いだけでは判定できなくなるからです。そのような状態遷移を「あいまいな状態遷移」と呼びます。 「選択疑似状態」を使うと、イベントの発生時に状況によって遷移先を振り分けることもできます。

状態遷移には、イベントが発生したとき、実際に状態を遷移するか判定する条件を追加できます。 この条件のことを「ガード条件」と呼びます。 ガード条件がある状態遷移では、イベントが発生したときにガード条件を評価します。 ガード条件の評価が真のときは、状態が遷移します。 ガード条件の評価が偽のときは、状態は遷移しませんが、その場合もイベントは起きたとことになります（「イベントが消費された」といいます）。

アクション

状態や状態遷移には、イベントが発生したときに実行したい処理を割り当てられます。 この処理のことを「アクション（またはエフェクト）」と呼びます。

状態遷移に割り当てられたアクションの場合、イベントが発生すると状態遷移に伴うアクションが実行されます。 状態が遷移するのは、アクションの実行が完了してからです。 ガード条件つきの状態遷移の場合は、イベントの発生後さらにガード条件も真なら、アクションが実行され、次の状態へ遷移します。

状態にアクションを割り当てる場合、次の3つの内部処理を記載できます。

状態内のアクション（内部動作）の種類
entry / 入場時アクション

その状態に入った直後に一度だけ実行される処理。

do / Doアクティビティ

状態に滞在している間、継続的に（または繰り返し）実行される処理。

exit / 退場時アクション

イベントが発生してその状態から遷移して抜けるとき、その直前に実行される処理。

    return s->attr_a && s->attr_b; // (2)
}

    return !s->attr_a || s->attr_b;
}

  1. ガード条件「gd1」判定用の関数。

  2. 属性値を使った条件式の例。

  3. 「gd2」用の関数。

ヒント
C言語での関数のスコープ

C言語では、関数名を static キーワードで修飾することで関数のスコープを制限できます。

  • static 宣言のない関数: 他のファイルから呼び出し可能（public相当）

  • static 宣言つきの関数: 同じファイル内からのみ呼び出し可能（private相当）

st0_proc などの内部処理関数は、 static 宣言つきの関数にして、外部から直接呼び出せないようにします。

3.6.8. 状態ごとの処理用関数を作成する（1）

状態ごとの処理を受け持つメソッドを作成します。 メソッド名は、状態名を小文字して「_proc」をつけるというルールにします。 したがって、状態名「ST0」の場合、関数名は「st0_proc」になります（ リスト 3.11 ）。

リスト 3.11 sample.c: 「ST0」の状態遷移の関数を作成する
static void st0_proc(Sample *s, Event evt, int p01) {
    switch (evt) {  // (1)
        case EV1:  // (2)
            if (gd1(s)) {  // (3)
                act1(evt, p01);  // (4)
                s->state.current = ST1;  // (5)
            } else {
                printf("Event ignored due to guard condition\n");  // (6)
            }
            break;
        case EV3:  // (7)
            act3(evt, p01);
            s->state.current = ST0;
            break;
        default:
            /* その他のイベントは無視 */
            break;
    }
}

  1. イベントで場合分けする。

  2. イベントが「ev1」の場合。イベントはenumのシンボルを使って表す。

  3. ガード条件による判定。

  4. アクション「act1」の呼び出し。

  5. 次の状態「ST1」への遷移。

  6. ガード条件が偽だったとき。イベントが無視されたことを表示する。

  7. イベントが「ev3」の場合。

リスト 3.15 sample.c
#include <stdio.h>
#include "sample.h"

/* ---------- State ---------- */

const char *state_name(State s) { // (1)
    switch (s) {
        case ST0: return "ST0";
        case ST1: return "ST1";
        case ST2: return "ST2";
        default:  return "UNKNOWN";
    }
}

/* ---------- SampleState ---------- */

void sample_state_init(SampleState *ss) {
    ss->current = ST0;
}

State sample_state_transit_to(SampleState *ss, State new_state) {
    if (new_state < ST_INVALID) {
        ss->current = new_state;
    } else {
        printf("invalid state: %d.\n", new_state);
    }
    return ss->current;
}

/* ---------- Event ---------- */

const char *event_name(Event e) { // (2)
    switch (e) {
        case EV1: return "ev1";
        case EV2: return "ev2";
        case EV3: return "ev3";
        default:  return "unknown";
    }
}

/* ---------- Sample ---------- */

void sample_init(Sample *s) {
    sample_state_init(&s->state);
    s->attr_a = true;
    s->attr_b = true;
}

static bool gd1(const Sample *s) {
    return s->attr_a && s->attr_b;
}

static bool gd2(const Sample *s) {
    return !s->attr_a || s->attr_b;
}

static void act1(Event evt, int prm) {
    printf("     act1: event:%s, param: %d\n", event_name(evt), prm);
}

static void act2(Event evt, int prm) {
    printf("     act2: event:%s, param: %d\n", event_name(evt), prm);
}

static void act3(Event evt, int prm) {
    printf("     act3: event:%s, param: %d\n", event_name(evt), prm);
}

static void st0_proc(Sample *s, Event evt, int p01) {
    switch (evt) {
    case EV1:
        if (gd1(s)) {
            printf("    gd1: true\n");
            act1(evt, p01);
            sample_state_transit_to(&s->state, ST1);
        } else {
            printf("    <<< gd1: false, transition is ignored. >>>\n");
        }
        break;
    case EV3:
        act3(evt, p01);
        sample_state_transit_to(&s->state, ST2);
        break;
    default:
        break;
    }
}

static void st1_proc(Sample *s, Event evt, int p01) {
    switch (evt) {
    case EV2:
        act2(evt, p01);
        printf("    gd2: %s\n", gd2(s) ? "true" : "false");
        if (gd2(s)) {
            sample_state_transit_to(&s->state, ST2);
        } else {
            act3(evt, p01);
            sample_state_transit_to(&s->state, ST0);
        }
        break;
    case EV3:
        act3(evt, p01);
        sample_state_transit_to(&s->state, ST0);
        break;
    default:
        break;
    }
}

void sample_play(Sample *s, Event evt, int p01) {
    printf("%s ->\n", state_name(s->state.current));
    printf("  event:%s, param: %d\n", event_name(evt), p01);

    switch (s->state.current) {
        case ST0: st0_proc(s, evt, p01); break;
        case ST1: st1_proc(s, evt, p01); break;
        case ST2: /* none */ break;
        default:  break;
    }

    printf("       -> %s\n", state_name(s->state.current));
    if (s->state.current == ST2) {
        printf("finished.\n");
    }
}

        Sample samp;

3.8.2. 検討用モデルに対応したプログラムを動かす

コマンドプロンプトを起動して（MacやLinuxならターミナルを起動して）、プログラムをビルドしましょう。 ビルドに使う Makefile は リスト 3.19 です。

リスト 3.19 sample03 で使用する makefile
CC      = gcc
CFLAGS  = -std=c99 -Wall -Wextra
TARGET  = sample
SRCS    = main.c sample.c sample_test.c
OBJS    = $(SRCS:.c=.o)

.PHONY: all clean

all: $(TARGET)

$(TARGET): $(OBJS)
	$(CC) $(CFLAGS) -o $@ $^

%.o: %.c
	$(CC) $(CFLAGS) -c $< -o $@

main.o:        main.c sample.h sample_test.h
sample.o:      sample.c sample.h
sample_test.o: sample_test.c sample.h sample_test.h

clean:
	$(RM) $(OBJS) $(TARGET)
コンパイラのオプションやマクロの意味

  • CC — コンパイラの指定（ここでは gcc を使うことを指定）

  • CFLAGS — コンパイラオプション用マクロ

    • -std=c99  — C99規格を指定

    • -Wall — 基本的な警告を有効化

    • -Wextra — 追加の警告を有効化

  • TAEAGET — 出力ファイル名を指定

  • SRCS — コンパイル対象のソースコードファイル

  • OBJS — SRCS の拡張子を .o に変えたもの

3.9. まとめ

クラス図とステートマシン図で動作を表現すれば、C言語のコードに変換できることがわかりました。

3.9.1. 構造のモデルを作成した手順

構造のモデルに登場する構成要素や要素間の関連を見つけ出し、スコアシートの構造を整理しました。

スコアシートの構造のモデルを作成した手順

  1. スコアシートに関する構成要素の発見

    • 具体的なスコアシートを観察して、そこにある要素を洗い出してオブジェクト図で表した。

  2. スコアシートに関するクラス図の作成

    • オブジェクト図に登場したオブジェクトからクラスを、オブジェクト同士のリンクから関連を見出し、クラス図に表した。

3.9.2. 構造のモデルだけではプログラムは作れない

構造のモデルを作ったことで、プログラムを構成する要素と、要素同士の関係（関連）を表せるようになりました。 また、クラスのインスタンスをオブジェクト図に見合うように作成すれば、実際に観察したスコアシートと同じようなデータ構造を作成できることもわかりました。

ところで、作りたかったのは、ボウリングのゲームスコアをつけるプログラムでした。 つまり、プレーヤーがフレームごとに交代して投球し、そのたびにスコアを更新する処理です。 ですが、これまでに作成した構造のモデルでは、ゲームの進行に応じてどんなことを処理するか表せていません。 そのためには、処理の中身を表す「振る舞いのモデル」を作成する必要があります。 次の章では、ステートマシン図を使って、フレームやスコアの振る舞いを表すモデルを作成します。
